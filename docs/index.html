FlyRank ML Internship — Capstone Research Page

Which content is about to decline — and can a rule-based flag actually catch it?

By Hafiz Muhammad Faizan · Machine Learning Track · 19 July 2026 · repo

Abstract. Content teams need to know which pages are worth reviewing before traffic drops further, but most flag-based rules are never checked against what actually happened next. Using one month (March 2026) of Google Search Console metrics from the FlyRank ML Internship warehouse — 3.6M daily GSC-available rows, 141,467 scorable content items — a transparent CTR-fix rule baseline was built first, then compared, on the same client-grouped held-out split and the same precision@K metric, to a Random Forest model trained on first-half-of-month features to predict whether a content item's clicks would fall in the second half. The Random Forest reached precision@50 of 0.90 against the rule baseline's 0.46 and a base rate of 0.25 on the same held-out clients; a leakage-detection test confirmed the evaluation harness itself works by showing the score jump to 1.00 when a deliberately leaked feature was added, then removed. The output is a ranked, human-reviewed action queue, not an automated publishing system.

01 · Introduction

The decision this supports

A content or SEO team managing hundreds of thousands of pages across dozens of clients can't manually review everything every month. They need a short, ranked, explainable list: which pages are worth looking at first, and roughly why. Get it wrong in one direction and editorial time is wasted on pages that weren't actually declining; get it wrong in the other and real decline goes unaddressed for another month. Neither failure is catastrophic on its own — which is exactly why this is built as a prioritization aid with mandatory human review, not an automated action system.

The concrete question: using only what's knowable in the first half of a month, can a model beat a simple, transparent rule at flagging which content items will see fewer clicks in the second half?

02 · Data

3.6M rows, one month, one table

Source: FlyRank/internship-warehouse (Hugging Face, gated), table fact_content_daily_performance — grain: report date × client × content item. This work uses March 2026 only, a mid-panel month iterated on directly rather than filtered down from the full 78.8M-row table. June 2026, the final month in the release, was deliberately never touched for label logic — it's the natural outcome window of any past-to-future label built on this data.

3,611,061 daily rows had gsc_data_available == True this month; after aggregating to one row per content item and requiring real impressions and position data, 141,467 content items had both features and a label, and 175,304 had usable position + impression data for the signal audit (a slightly larger set, since the audit doesn't require a second-half label).

Columns used

gsc_impressions, gsc_clicks, gsc_avg_position, gsc_data_available.

Deliberately excluded

  • GA4 columns — zero-filled before each client's GA4 tracking start date; out of scope for this GSC-only lane.
  • Session-source and AI-channel columns — describe traffic mix, not search visibility or click-through; a different lane's problem.
  • Client and content hash IDs — pseudonyms, used only for grouping and for the train/test split, never as a model input.
  • Any second-half-of-month aggregate as a feature — that window is reserved for the label alone. Feeding it in as a feature is the exact leak tested for and removed in Section 3.

Public-safe by construction: every ID in this dataset and this page is a pseudonym (e.g. client_23a62021009f63c4). No client names, raw URLs, or query text appear anywhere in this project.

03 · Methodology

A rule first, a model second, one honest split for both

The baseline: a transparent rule

Content already ranking decently (position 1–20) whose click-through rate falls short of its own position bucket's typical CTR that month is flagged with one reason code — CTR_FIX_OPPORTUNITY — scored by the estimated extra clicks available if CTR caught up. It's a fair baseline precisely because it uses the same single month, the same table, and no future data — nothing the model gets that the rule doesn't.

The label

is_declining_label = 1 if a content item's total clicks in days 16–31 were lower than its total clicks in days 1–15 of the same month, else 0. Base rate across the full scored population: 20.3% declining.

The features (all knowable at day 15)

Clicks, impressions, and CTR from the first half of the month; average position (with zero-position "no data" rows excluded before averaging, not treated as rank one); a position-bucket category; and a count of active days. Nothing from days 16–31 appears anywhere in the feature set.

The model

Logistic Regression, then Random Forest — chosen because the target is a genuine yes/no observed label, which maps directly to that path on the project's own method-selection table (readable first, stronger second, kept only because it earned the comparison below).

Validation design

Split grouped by client (75/25 by client count: 32 clients train, 11 clients test), not a random row split. Content items from the same client share hidden characteristics — a random split would let a model partially memorize client-specific patterns. Verified zero clients appear in both train and test.

Leakage checks

Ran the full attack checklist: timeline drawn (features strictly before the label window), a scan for label-derived or sibling columns in the features (none found), a scan for product-flag columns as features (none — the rule's own score is used only as a baseline to beat, never as a model input), and a deliberate-leak test — adding the second-half click aggregate directly and confirming the score inflated before removing it (Section 4).

Random Forest beat both the rule baseline and Logistic Regression at every K tested. The rule baseline itself cleared the base rate by roughly 2x at precision@50 (0.46 vs 0.25) — the transparent rule has real, if modest, signal on its own before any model is involved.

Before/after the honest split

The same Random Forest, trained twice — once on a plain random split, once on the client-grouped split above:

DIRECTIONAL Counter to the usual expectation, the grouped split scored slightly higher here, not lower — the random split showed no evidence of client-level memorization inflating its number. The most likely explanation is that the 11 held-out clients in the grouped split happened to have a higher base rate (25.2% vs the population's 20.3%), which mechanically makes precision@K easier to hit regardless of the model. With only 11 clients in the test group, this comparison itself is noisy — a single unusual client can move it. This is reported as-is rather than reframed to fit the expected story.

Signal audit — do the assumptions behind the rule actually hold?

Each verdict below is computed from a real Spearman correlation test on 175,304 content items this month, not asserted:

  • CONFIRMED CTR vs. position — the CTR-fix flag's core assumption. Mean CTR falls from 1.10% (position 1–3) to 0.10% (position 51+); Spearman ρ = -0.254, p < 0.0001.
  • OPPOSITE Impression volume vs. CTR — the hypothesis was that high-volume content would show lower CTR (diluted relevance at scale); the data showed the reverse (ρ = +0.596, p < 0.0001) — higher-volume content actually had higher CTR in this slice.
  • CONFIRMED Position-tracking consistency vs. CTR — more consistent day-to-day position data associated with higher CTR (ρ = +0.219, n = 175,304).
  • CONFIRMED Weekday vs. weekend click volume — weekend clicks ran 8.7% lower than weekday clicks.

Reading the errors

Permutation importance (checked by shuffling, not read off raw coefficients) named the top three features as ctr_first_half (0.062), clicks_first_half (0.057), and active_days_first_half (0.038) — none towered suspiciously above the rest, a check against a missed leak. Accuracy by position bucket was weakest in the 1–3 bucket (71.4% accuracy, n=1,038) — the best-ranking content was actually the hardest to call correctly, likely because top-position content's CTR swings are noisier relative to its usual baseline. The most confidently wrong cases shared a pattern: very thin first-half impressions (as low as 5) driving an unstable CTR estimate that the model over-trusted.

Verifying the test harness itself

Deliberately added the second-half click aggregate as a feature: precision@50 jumped from 0.900 (honest) to 1.000 (leaked) — confirming the evaluation harness correctly detects leakage when it's present. That feature is not in the shipped model; 0.90 is the number that ships.

05 · Limitations & honest framing

What this work cannot claim

  • DIRECTIONAL One month (March 2026) only — not validated across other months, seasons, or client verticals.
  • One data source — Google Search Console metrics only. Says nothing about content quality, backlinks, technical SEO, or GA4 engagement.
  • Cross-sectional, not experimental — no content was actually rewritten and measured here. The claim ladder stops at "these items look worth reviewing first, because…", not "doing this will produce that."
  • The grouped-split test group is only 11 clients — small enough that the before/after comparison in Section 4 should be read as suggestive, not conclusive.
  • Thin-history clients and low-position-coverage items (45,448 of 141,467 items) are explicitly excluded from ranking rather than scored with false confidence.
  • This does not model, predict, or claim anything about how any search engine's ranking algorithm works — it describes one portfolio's observed click pattern in one month.

What must never be automated

DECISION-SUPPORT No automatic publishing of rewritten titles, meta, or refreshed content — a human writes and approves every change. No bulk content-refresh actions without editorial sign-off — it's the most expensive, highest-risk tier in this queue (4,511 items). No treating the model's score as a calibrated real-world probability; it's a ranking signal for one month, not a forecast.
